# Universal code style principles

These principles apply across every language and format Tianon writes, wherever the target permits them.  Each language-specific doc notes how the principle applies (or why it cannot) in that context.

## Indentation: hard tabs

Hard tabs are the universal indentation character for all code Tianon writes.  This applies to bash, jq, Go (enforced by `gofmt`), Perl, AWK, Dockerfile continuation lines, and anywhere else the format permits.

The reasons: tabs look cleaner, use fewer bytes, and let every reader configure their editor to the tab-stop width they prefer.  They are also easier to type manually when needed.

Forced exceptions — formats where tabs are prohibited by spec or ecosystem convention:

- **YAML**: the YAML specification explicitly forbids tab characters in indentation.  2 spaces are used — not because that is preferred, but because the spec leaves no choice.
- **Python**: the Python ecosystem and tooling effectively prohibit hard tabs.  Python 3 rejects mixed indentation, and the community convention (`PEP 8`) is 4 spaces; tabs look wrong in essentially every Python context.

**JSON is not an exception**: JSON files are generated by `jq --tab`, which produces tab-indented output.  This is the correct and preferred form.  The common 2-space JSON default seen in many tools is not Tianon's preference.

## Inline vs multi-line: the clarity threshold

When choosing between inline and multi-line layout for any construct in any language, the question is: **is this short *and* clear on a single line?**

There is no hard rule — it is vibes-based, in the spirit of Linus Torvalds's "the numbers got too big."  Corpus analysis suggests approximately 60 characters as a rough guide, but length alone is not the criterion: a short expression that is hard to parse at a glance still gets multi-line treatment.

This applies uniformly across jq `if`/`else`, shell test expressions, function call argument lists, Dockerfile `RUN` command chains, and anywhere else a similar choice arises.

## Vim modelines

Files with non-standard extensions or no extension that vim cannot auto-detect carry a vim modeline comment to set the filetype explicitly:

```bash
#!/usr/bin/env hocker
# vim:set ft=sh:
```

```
# vim:set ft=gitconfig:
```

```nginx
# vim:set ft=nginx:
```

Vim checks the first and last few lines of any file for modelines, so placement is flexible.  The guiding principle is **least disruptive and least likely to be accidentally removed**:

- **Executable scripts**: second line, immediately after the shebang — stays at the top where edits to the body are very unlikely to touch it
- **Non-executable config files**: last line — the start of a config file often carries meaningful content that should not be preceded by a comment; the end of the file is safe and conventional

Corpus refs: [`home/git-config.d/common#L1`](https://github.com/tianon/home/blob/720c476e79a50ab0dd133f7187bd046b32cd5b73/git-config.d/common#L1) (first line), [`squignix/http.conf`](https://github.com/tianon/squignix/blob/fe9946c06ac4f9163ed81f2ba499835dc0494097/http.conf) (last line).

## Trailing commas in multiline structures

Wherever a format permits trailing commas, prefer them.  This makes diffs cleaner (additions don't touch the preceding line) and removes a class of "forgot the comma" errors.

Language-specific applications:

- **jq objects**: always include a trailing `,` after the last field in a multiline object literal
- **jq generators**: use `empty` as a final "no-op" element to allow all real elements to have trailing commas — `empty` generates nothing but lets you write `"last-item",` and keep the comma (see [jq.md](jq.md))
- **Go**: `gofmt` requires trailing commas in multiline function call argument lists; this is enforced automatically
- **Python**: trailing commas in multiline function calls, lists, tuples, and dicts are preferred (Tianon uses Python rarely, but the same principle applies)
- **JSON**: trailing commas are forbidden by spec — not applicable
- **YAML**: not applicable (sequence entries don't have separators)

The specific mechanics in each language are documented in that language's style doc.
